# 전체 기획 흐름

`product-blueprint:orchestrate`는 모든 문서를 혼자 쓰는 스킬이 아니라 단계별 전문 스킬을 연결하고 사용자 결정을 기록하는 진행 관리자입니다.

## 1. 아이디어와 근거

`brief` → `research` 또는 `ideation` → 필요 시 `reference-deconstruction`

- 알고 있는 사실, 사용자에게 들은 내용, 제안, 가정, 확인하지 못한 내용을 구분합니다.
- 로그인 뒤처럼 직접 확인하지 못한 레퍼런스 흐름은 추측으로 채우지 않습니다.

## 2. 제품과 브랜드 방향

`parallel-concepts` → 사용자 제품 방향 확정 → `positioning-brand` → 사용자 브랜드 방향 확정

- 제품 구조와 핵심 행동이 다른 방향을 비교합니다.
- 이름 검색 결과는 충돌 가능성을 살피는 신호일 뿐 상표 검토가 아닙니다.

## 3. PRD와 교차점검

`experience-mechanisms` → `prd` → `planning-quality-gate` → 사용자 첫 버전 범위 확정

교차점검은 제품전략, 사용자근거, 브랜드, PRD, 서비스·실현성, 성장·리스크의 여섯 관점으로 진행합니다. 중요한 문제가 해결되기 전에는 다음 단계로 넘어가지 않습니다.

## 4. 제품 정의와 화면 계약

`product-definition` → `screen-contract` → `service-contract` → 초기 `feasibility-review`

- 사용자, 진입점, 여정과 필요한 결과를 먼저 확정합니다.
- 그 정의에서 화면의 목적, 행동, 상태와 복구 방식을 파생합니다.
- 화면 뒤 서비스 책임과 제품에 드러나는 제약을 기록합니다.

## 5. 흐름과 디자인 전달

coverage self-audit → `storyboard` → Standard/Deep의 `backend-systems-brief` → `design-brief`

스토리보드는 화면 간 행동을 확인하는 낮은 해상도의 흐름 보드입니다. 색, 글꼴, 컴포넌트 스타일이나 완성 화면을 결정하지 않습니다. 시스템 동작이 단순한 제품의 백엔드·시스템 브리프는 짧게 작성하고 해당하지 않는 항목의 이유를 남깁니다.

`03-design-brief.md`가 만들어지고 기획 검증이 통과하면 기본 과정은 끝납니다.

## 사용자가 직접 결정하는 네 지점

1. 제품 방향
2. 브랜드 방향
3. 첫 버전의 포함·제외 범위
4. 사용자, 진입점, 주요 여정과 필요한 결과

이 결정은 `00-decision-log.md`에 기록합니다. 사용자에게는 위 표현을 사용하고, 데이터 파일 안의 상태 코드나 검증 용어를 그대로 노출하지 않습니다.

## 다시 실행할 때

- 제품 방향을 바꾸면 브랜드, 메커니즘, PRD와 이후 정의를 다시 점검합니다.
- 브랜드만 바꾸면 PRD의 언어와 디자인 전달 문서를 다시 점검합니다.
- 첫 버전 범위를 바꾸면 제품 정의, 화면·서비스 계약과 흐름을 다시 점검합니다.
- 화면 디자인 중 제품 행동이 바뀌면 디자인 안에서 처리하지 말고 해당 기획 단계로 돌아옵니다.

## 기본 과정이 하지 않는 것

- 최종 색상, 글꼴과 컴포넌트 스타일 결정
- 고해상도 UI 화면 일괄 생성
- 완성 프로토타입과 디자인 승인
- API, DB, 프런트엔드·백엔드 아키텍처 결정

이 중 시각 디자인과 프로토타입은 사용자가 원할 때 `product-blueprint:design-production`에서 별도로 진행합니다.
