# 선택형 디자인 제작 과정

`product-blueprint:design-production`은 기획이 끝난 뒤 사용자가 명시적으로 요청할 때만 시작하는 별도 과정입니다.

## 시작 조건

- Standard 또는 Deep 프로젝트에서 `--stage planning`이 `planning-structure-pass`로 통과했습니다.
- `03-design-brief.md`가 현재 기획과 일치합니다.
- 사용자가 화면 디자인 또는 프로토타입 제작을 원하며, 그 선택이 `design-entry` 결정으로 기록됐습니다.

필요한 파일을 기존 기획에 추가합니다. 기존 문서는 덮어쓰지 않습니다.

```bash
python3 scripts/init_prd_project.py "<제품 이름>" --root <기획 루트> --with-design
```

## 진행 순서

1. 사용자가 좋아하는 것과 싫어하는 레퍼런스를 확인합니다.
2. `visual-directions`로 제품에 맞는 시각 방향 2~3개를 같은 수준으로 비교합니다.
3. 사용자가 하나의 방향을 선택하고 `visual-direction` 검증을 통과합니다.
4. `key-screen-exploration`으로 대표 화면 하나와 주요 상태를 좁고 넓은 화면에서 만듭니다.
5. 사용자 피드백을 반영해 대표 화면을 승인하고 `key-screen` 검증을 통과합니다.
6. 승인된 방향을 디자인 시스템과 나머지 화면에 확장합니다.
7. 프로토타입으로 주요 흐름과 실패·복구 상태를 테스트합니다.
8. 현재 증거를 사용자가 승인한 뒤 디자인 전달 문서를 만듭니다.

`design-pass`는 현재 디자인 기준선 승인까지만 뜻합니다. 별도의 전달 문서와 `handoff-pass`가 있어야 제품·디자인 전달 준비로 표시됩니다.

## 품질 원칙

- 한 번에 모든 화면을 생성하지 않습니다.
- 대표 화면의 품질이 승인되기 전에 전체 화면으로 확장하지 않습니다.
- 전용 프런트엔드·디자인 역량과 충분한 레퍼런스가 없으면 한계를 밝힙니다.
- 스토리보드, 생성 이미지, 첫 렌더링을 완성 디자인으로 부르지 않습니다.
- 디자인이 제품 행동을 바꾸어야 한다면 PRD와 화면·서비스 계약으로 돌아가 다시 확인합니다.
- 상위 결정이 바뀌면 해당 지점부터 workflow gate를 무효화하고 중간 승인과 증거를 다시 만듭니다.
- 에이전트의 자체 평가가 사용자 승인을 대신할 수 없습니다.

## 결과물

- 선택된 시각 방향과 근거
- 승인된 대표 화면과 상태
- 재사용 가능한 디자인 규칙과 컴포넌트
- 전체 핵심 화면과 반응형 증거
- 테스트된 클릭형 프로토타입
- 사용자 승인 기록과 제품·디자인 전달 문서

이 과정 역시 실제 구현 품질을 보증하지 않습니다. API, 데이터베이스와 애플리케이션 아키텍처는 이후의 별도 기술 설계에서 결정합니다.
